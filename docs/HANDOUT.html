Dokumentasi

Template PRD: Project Requirements Document

AI akan membangun apa yang Anda tulis. Jika Anda tidak menulisnya, AI tidak akan membangunnya. Template ini memastikan tidak ada yang terlewat.

⚠️ Mengapa PRD Ini Berbeda?

Kebanyakan PRD hanya berisi functional requirements (fitur apa yang dibuat). PRD ini memaksa Anda mendefinisikan non-functional requirements: berapa user simultan? berapa uptime? apa ancaman security? Tanpa ini, AI akan membangun sistem yang bekerja di laptop Anda dan runtuh di produksi.

1. Problem Statement

  • Apa masalah bisnis yang diselesaikan?
  • Mengapa sekarang?
  • Seperti apa suksesnya?

2. Functional Requirements

  • Apa yang harus sistem LAKUKAN?
  • User stories: Sebagai [user], saya ingin [fitur], sehingga [manfaat]
  • Prioritas: Must have / Should have / Could have

3. Non-Functional Requirements (NFR) — PENTING!

  • Performance: Berapa concurrent user? Berapa RPS? Response time target?
  • Availability: Berapa % uptime? (99.9% = 8.7 jam downtime/tahun)
  • Security: Metode autentikasi? Role-based access? Klasifikasi data sensitivitas?
  • Scalability: Load saat ini? Pertumbuhan yang diharapkan?
  • Maintainability: Skill tim? Dokumentasi yang dibutuhkan?

4. Constraints & Assumptions

  • Harus bekerja dengan MySQL yang ada?
  • Budget terbatas?
  • Tim hanya 2 developer part-time?
  • Data tidak boleh keluar dari jaringan universitas?

5. Acceptance Criteria

  • Bagaimana kita TAHU ini selesai?
  • Testable: "API mengembalikan student count dalam < 2 detik"
  • Security: "Attempt DELETE request menghasilkan 405 error"

6. Risk Register

  • Database performance menurun saat peak
  • Akses data tidak sah (no auth)
  • Tim kurang skill Python
  • AI memberikan jawaban salah
  • Scope creep (tambahan fitur tanpa kontrol)
Peringatan

Cautionary Tales: Ketika Agentic Workflow Salah

AI tidak menghilangkan engineering yang buruk. AI membuat engineering buruk menjadi lebih cepat. Cerita-cerita ini nyata dan terjadi di industri.

💥

Tale 1: "Works on My Machine"

API dibangun AI bekerja sempurna dengan 10 data test. Saat 5,000 mahasiswa mendaftar bersamaan, sistem lock-up. Database SQLite single-writer tidak sanggup.

Pelajaran: Tentukan load requirement di PRD. Load test sebelum produksi.
🗑️

Tale 2: Agent yang Menghapus Data

AI assistant diberi akses tulis. Kepala departemen bilang: "Hapus mahasiswa yang belum bayar." Agent menghapus 200 record. Tidak ada backup. Tidak ada audit trail.

Pelajaran: API baca-saja dulu. Tulis butuh approval manusia.
🔓

Tale 3: API "Aman" Tanpa Auth

API internal "diamankan" dengan firewall. Mahasiswa menemukan endpoint dan mengunduh 10,000 record data pribadi. Data breach.

Pelajaran: Jika tidak ada auth, itu publik. Internet sangat pandai menemukan.
🔒

Tale 4: Vendor Lock-In

Universitas membangun seluruh workflow di Claude. Anthropic ubah harga. Tidak bisa pindah karena semua integrasi kustom. Terjebak.

Pelajaran: Gunakan standar (MCP). Test dengan multiple agents.
🎭

Tale 5: "Vibe Coded" Production

Portal mahasiswa dibangun AI dalam semalam. 3 bulan kemudian: tidak ada yang mengerti strukturnya, tidak ada test, developer asli sudah resign.

Pelajaran: AI menulis kode. AI tidak memiliki kode. Anda yang memiliki dan memeliharanya.
📋

Tale 6: PRD yang Tidak Ada

"Bangun sistem registrasi." AI membangun sesuatu. Punya fitur chatbot yang tidak diminta. Tidak bisa export ke finance. Setelah 3 bulan, dibuang.

Pelajaran: AI adalah implementor brilian dan product manager buruk. Anda harus jadi product manager.
🎯 Meta-Lesson

AI tidak membuat engineering buruk hilang. AI membuat engineering buruk lebih cepat. Tujuan workshop ini bukan membangun lebih cepat. Tujuan adalah membangun dengan BENAR, dengan AI sebagai alat, bukan pengganti berpikir.
